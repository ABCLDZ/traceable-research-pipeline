============================================================
搜索工作流 — 最终项目报告
============================================================

一、项目总结
============================================================

从零搭建了一套"搜索工作流"——覆盖公开资料采集、HTML/PDF解析、
文本分块、LLM证据抽取、原文验证、结构化聚合和研报分析的完整管线。
全部使用真实 DeepSeek API 运行。

总成本约 ¥6（人民币），已跑通 5 个真实文档、3 个实验场景。

二、实验清单
============================================================

实验一：real_trial_01（美国AI数据中心电力风险）
  材料: EIA HTML(9.7KB) + NERC PDF(606KB,253页) + DOE PDF(125KB)
  结果: 55 张证据卡
  版本: V1（直接输出原文）
  发现: 24% 精确匹配率 + 缓存bug导致 26% 重复调用
  成本: ¥5（含重复计费）

实验二：V1→V2→V3 迭代
  V1: 模型直接输出原文 → 24% 精确匹配
  V2: 模型选句子ID → 100% 匹配但 Prompt太简导致大量空响应
  V3: 定性与定量混合 + 截断宽容解析 → 100% 匹配且稳定产出
  关键修复: 磁盘缓存 / JSON brace-counting / 口径归一化

实验三：公开上市公司季度公告 + 年报
  材料: 季度公告(21KB,8chunk) + 年报(803KB,318chunk)
  结果: 214 张证据卡 (季度公告=118定量 + 年报=96定性)
  精确匹配: 100%
  总成本: ¥1.08（季度公告=¥0.06 + 年报=¥1.02）
  说明: 公司名称、文档标识和具体财务结论已从开源版本移除。

六、经验教训
============================================================

1. 模型改写原文不是提示词问题，是架构问题
   让生成模型同时做语义理解 + 逐字引用是冲突的。
   解决方案：span-selection 协议（模型选ID，代码拼 excerpt）。

2. 磁盘缓存是 V1 的核心教训
   无持久化缓存的 batch 脚本每重启一次就重复计费。
   26% 的重复调用贡献了约 ¥3.7 的浪费。

3. PDF 表格损失不可逆
   pdfplumber 提取后表格结构丢失，行列关系无法恢复。
   iXBRL 格式（SEC 20-F）需要专用解析器。

4. 证据膨胀需要分层
   一个财报生成 118 张卡，但其中多数是资产负债表科目级别的
   AtomicFact。需要聚合为 ResearchEvidence 后才能进入研报端。

5. 小样本诊断比全量跑更高效
   NERC V2 全量跑 253 chunk 花了约 ¥1 的无效调用，
   而 NERC V3 40 个样本就足够判断效果。
